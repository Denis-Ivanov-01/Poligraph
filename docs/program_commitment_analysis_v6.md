# Program commitment analysis workflow

## Selected architecture

The section operation is one persisted parent `AiRun` with one child `AiRun` per commitment in the selected scope. A section is not capped at 50 commitments. The default value of 50 is only an internal tranche size used to generate manageable prompts and imports.

Every tranche uses the same canonical methodology as standalone commitment analysis. The parent batch stores the full expected commitment set, while each tranche prompt asks the model to return analyses only for the currently pending commitment refs. This keeps the manual web flow simple while preserving item-level validation and partial retry support.

The repository still has a manual execution boundary: the app generates prompts and imports JSON pasted by an operator. A future model executor can use the same action layer, prompt builders, validators, and import logic without changing the internal web flow.

Model output JSON is intentionally content-only. Run provenance such as model name/model type, prompt version, schema version, methodology version, task type, target binding, analysis date, token counts, and tool counts is stored on `AiRun` and returned in the CLI/action response envelope instead of being generated by the model.

Prompt JSON contracts are generated from the SQLAlchemy model columns where the field is persisted, and enum strings are rendered from the shared application constants. This keeps prompt examples, validators, and database-backed fields aligned when the model schema or enum vocabulary changes.

## Prompt composition

`commitment_analysis_methodology.py` is the canonical source for both standalone and tranche-based section analysis. The initial classification prompt deliberately withholds previous status summaries and section narratives so the model is not anchored to outdated assumptions.

The model is asked to reason through the methodology for each commitment independently, including material components, baseline, authority, implementation evidence, contribution, contradictory evidence, current figures, and confidence calibration. During structural extraction the model assigns `importance_level`; the backend derives the numeric `importance_weight`. The backend also owns deterministic values such as status grouping, review flags, expected refs, duplicate checks, source/link integrity, and section readiness.

Section summaries are generated only after every commitment analysis in scope has been imported successfully.

## Internal web flow

The internal app keeps the original simple pages:

1. Open the batch status action from the section/program preview.
2. See a single generated prompt for the next tranche.
3. Save the prompt and run it externally.
4. Paste the JSON response on the JSON import page.
5. If commitments remain, the app automatically opens the next tranche prompt.
6. When all child analyses are imported, the app returns to the program preview.

For example, a section with 123 commitments is represented by one parent batch with 123 child runs. Internally it is processed as 50 + 50 + 23 by default, not as three separate section analyses.

## Database ownership

The database keeps root-methodology and operational workflow fields only. Prompt-only reasoning structures such as material component breakdowns, evidence gaps, review reason lists, and component-scoped evidence claims are validated at import time but are not persisted as first-class commitment columns.

Migration `0016_revert_v6_schema` removes the v6 prompt-only analytical persistence added by the previous experimental workflow. Operational parent/child `AiRun` fields remain because they are needed for full-section batches, progress tracking, retries, and future executor integration.

## CLI boundary

The CLI uses the same workflow action layer as the internal web app. It can generate standalone prompts, import standalone status JSON, start a section batch, inspect a batch, import child analysis JSON, approve structural changes, and publish a program. Future AI integrations should plug in below this boundary rather than duplicating prompt, validation, import, or publication logic.

## Validation

Validation checks exact commitment coverage for the current tranche, source references, enums, contribution explanations, and consistency rules before importing. Successful child analyses are kept when a later tranche fails, so the operator can retry the failed tranche without losing earlier work.
